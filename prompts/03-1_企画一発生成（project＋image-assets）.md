# 03-1_企画一発生成（project.md＋image-assets.md）

`03`（project.md化）→ `04`（スライド構成・原稿・設計）→ `05`（画像アセット策定）の **3工程を1ターンで畳む**、ChatGPT 用の一発生成プロンプトです。

## 用途・前提

- **壁打ち（01〜02）を済ませた同じチャットの続き**で投げる前提です。壁打ち文脈がチャットに残った状態で使ってください。
- 質を死守したい場合は使いません。これは **中間（03〜05）だけ時短したいとき**の代替ルートです。
- このルートでは **画像案の採用判断（通常 05 の Part A→人間判断→Part B）も AI が一任で行います**。つまり「画像を何枚・どの役割で使うか」を人間がレビューしません。
  - そのぶん、**生成画像を目で見る**チェック（06-1 のあと）は省略不可です。画像ラインナップの良し悪しはそこで初めて人間が判断します。
- 出力は **`project.md` と `image-assets.md` の2ファイル**。`project.md` に 04 のスライド構成・原稿・画面設計を **section 単位で内包**させます。

## 後続フロー

```text
01〜02 壁打ち（ChatGPT・死守）
  ↓
03-1 一発生成（ChatGPT続き）→ project.md ＋ image-assets.md
  ↓
👁 ファイル2つを目視
  ↓
06-1 Codex 一括画像生成 → 👁 生成画像を目視 →（ダメなら07で改善）
  ↓
08 Claude Code 実装 → 09 Codex レビュー
```

## プロンプト本体

```text
このチャットでここまで壁打ちして決めた内容を正本として、
次工程に渡せる「project.md」と「image-assets.md」を一度に作成してください。

目的:
このあと Codex で画像生成、Claude Code で HTML化、Codex でレビューを行います。
過去会話を読み返さなくても、この2ファイルだけで制作を進められる状態にしてください。

前提:
- このチャットで確定した結論を正とする。確定していない過去案は確定事項として扱わない。
- 未確定の情報は推測で埋めず「要確認」と明記する。
- 不明点だけで止まらず、仮定を置ける範囲は仮定を置いて進める（仮定はその旨を書く）。

重要方針（画像とデザインの基本）:
- 最終成果物は画像付きのHTMLスライド資料。section 単位で見せ場が分かれる構成にする。
- 1 section = 1メッセージを基本にする。
- 画像内文字は、短いタイトル・コピー・ラベル・UI文言として必要に応じて入れる。
- 長文説明、細かい注釈、発表時の補足、あとから変更されやすい文章はHTML側に残す。
- すべての画像に無理に画像を入れない。画像は装飾ではなく、理解・印象・説得力を強めるために使う。
- 表紙は雑誌表紙・ポスター・キャンペーンキービジュアルのように強めにする。
- UI説明は説明性を残したまま少し華やかにする。導線説明は分かりやすさとグラフィック性を両立する。
- 世界観画像は単なる背景ではなく、提案の印象を強める画像として扱う。
- 普通のPowerPoint風に寄せすぎない。ただし提案資料として読みにくいほど派手にしない。
- 画像内文字とHTML側見出しを同じ強さで二重表示しない。
- 画像枚数は増やしすぎない。1画像1役割を基本にし、似た構図を連続させない。
- 人物・キャラクターは必要な画像にだけ使う。
- 指定外の文字、意味不明な文字、不要な英字、架空のロゴ文字は入れさせない（画像生成promptで必ず禁止する前提で書く）。

このターンでやってほしいこと:
1. 壁打ち結論を、制作仕様書 project.md にまとめる。
2. その中に、04相当のスライド構成・原稿・画面設計を section 単位で内包させる。
3. 各 section に対して、画像が必要か／画像なしでよいかを自分で判断する。
4. 必要な画像について、役割・構図・比率・画像内文字を自分で取捨選択して確定する（採用判断もこのターンで行う）。
5. 確定した画像を image-assets.md にまとめる。

出力フォーマット:
- 出力は「project.md」と「image-assets.md」の2ファイル。
- 各ファイルは、そのまま保存できるダウンロード形式（タップ／クリックで保存できる形）を第一にする。
- ダウンロード形式で出せない環境・モードの場合のみ、フォールバックとして各ファイルを1つのMarkdownコードブロックで出力する（2ファイルなのでコードブロックは2つ）。
- ファイル名は「project.md」「image-assets.md」とし、「-main」などの接尾辞・サフィックスを付けない。
- ファイルやコードブロックの外に前置き・補足を書かない。

==== project.md の構成 ====
## 1. 全体概要
## 2. 最終的に作りたいもの
## 3. 想定読者・使用場面
## 4. 確定した方針
## 5. 仮決定・保留事項
## 6. 採用しない方針
## 7. デザイン・トーンの方向性
## 8. 実装・制作上の条件
## 9. スライド構成（section単位）
   各 section について以下を書く:
   - section番号
   - sectionタイトル
   - このsectionの役割
   - 見せたいメッセージ
   - 画面に載せる短い文章（HTML側テキスト）
   - 発表時に話す補足
   - 画像・ビジュアルの使い方（画像あり/なし、役割、比率、配置方針）
   - レイアウト方針
   - 注意点
## 10. 未確定事項・確認が必要なこと

project.md の条件:
- 会話で確定したことと未確定のことを分ける。採用しない方針も明記する。
- project.md の文章をそのまま section にコピーしすぎない。HTML化しやすい section 単位にする。
- 完成版として固めすぎず、70〜80%のたたき台でよい（後で壁打ちで足せる）。

==== image-assets.md の構成 ====
## 1. 概要
- この資料の目的 / 画像の使い方の基本方針 / 画像内文字の基本方針 / デザイン温度の基本方針 / HTML化時に守ること
## 2. 画像一覧表
   各画像について以下を表で整理:
   - リネーム後のファイル名（例: 01_cover.png）
   - 使用セクション / 画像の役割 / 画像タイプ / デザイン温度 / 見た目の方向性 / 主役要素 / 推奨比率
   - 画像内文字の有無 / 画像内に入れる文言 / HTML側に残す文言 / 人物・キャラクターの有無
   - HTML上での配置方針 / alt属性の案 / 注意点
## 3. 各画像の詳細仕様
   画像ごとに: ファイル名 / 使用section / 目的 / 画像タイプ / デザイン温度 / 見た目の方向性 / 主役要素 /
   画像で見せる要素 / HTML側で補足する要素 / 画像内に入れる正確な文言 / HTML側で重複させない文言 /
   推奨レイアウト / object-fit・object-position の注意 / 人物・キャラクターの扱い / 避けたい使い方 /
   画像内文字の検品ポイント / alt属性案
## 4. 画像内文字の共通ルール
## 5. HTML化時の共通ルール
## 6. Claude Code / Codex への引き継ぎメモ

image-assets.md の条件:
- 画像を増やしすぎない。1画像1役割。似た構図を連続させない。
- 画像内文字は短く、具体的な文言まで書く。日本語を基本にする。
- 画像内文字とHTML側テキストが重複しすぎないようにする。
- デザイン温度（強め/中程度/説明寄り）と見た目の方向性を各画像に必ず設定する。
- 06-1（画像生成）で必ず守る条件を、画像ごとの「注意点」に反映する。
- 未確定は「要確認」と書く。

入力情報:
---
【このチャットの結論を使う】
このチャットでここまで決めた内容を正本としてください。
（別途まとめがある場合のみ以下に貼る）
-

【特に優先したい方針・変更点】
-

【使いたいキャラクター・参考画像】
-

【避けたい表現】
-
---
```

- `project.md` に企画仕様と section 単位の構成・原稿・画面設計が内包されている。
- `image-assets.md` に画像ごとの役割・配置・画像内文字・HTML側文言・デザイン温度が決まっている。
- 2ファイルとも、そのまま保存できる形式（ダウンロード優先、不可時はMarkdownコードブロック）で出力されている。
- 未確定事項が「要確認」として残っている。
- このあと **06-1 の生成画像目視**で、画像ラインナップを人間が初めて確認する流れになっている。
