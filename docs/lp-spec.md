# 新LP仕様書: AI HTMLスライド制作プロンプト集

## 1. 目的

このLPは、HTMLスライド生成プロンプト集を実際に使いたい人に向けて、使い方、工程、成果物例をわかりやすく伝えるためのページである。

発表用スライドではなく、閲覧者が上から読んで「何ができるか」「どの順番で使うか」「どんな成果物になるか」を理解できる広告LP型の縦スクロールページにする。

## 2. 最優先ユーザー

最優先する対象は、promptを実際に使う人。

想定する閲覧者は以下。

- AIでHTMLスライドを作りたい人
- 企画、構成、画像、HTML化、レビューの流れを整理したい人
- どの `.md` をどの順番で使うか知りたい人
- 成果物例を見て、自分でも使えそうか判断したい人

## 3. 基本方針

- スライド形式ではなく、縦スクロールLPにする
- prompt本文はLP内に全文掲載しない
- GitHub上のpromptファイルへリンクする
- 活用マップ、最短ルート、promptカードで使い方を伝える
- 成果物プレビューはファーストビュー画像つきカードで見せる
- 初期表示は代表成果物だけにし、もっと見るボタンで全件表示する
- スクリーンショット自動撮影は内部運用とし、公開LPの説明セクションにはしない
- Portfolio用途は考えない
- `slide-archive` とは統合しない
- `slide-archive` のURLは、必要な成果物リンクとして参照するだけにする

## 4. ページ構成

### 4.1 ファーストビュー

目的は、何のLPかを一瞬で伝えること。

表示内容。

- ヒーロー画像: `assets/images/hero-main.png`
- 画像内タイトル: AIと作るHTMLスライド制作プロンプト集
- 画像内サブコピー: 企画からHTML化まで、流れで使える制作ワークフロー
- CTA: 成果物を見る、使い方を見る

ヒーローは一枚絵を主役にする。工程カードやワークフロー図は置かない。

### 4.2 できること

このprompt集で可能になることを短く示す。

- 企画を整理する
- スライド構成を作る
- 画像仕様を決める
- HTML化する
- Codexでレビューする

### 4.3 使い方マップ

`00_作業の流れ.md` をベースに、工程の順番を図解する。

全文を読ませるのではなく、入口として機能させる。

### 4.4 初めて使う人向け最短ルート

全promptを最初から読ませず、まず使うべきファイルを絞って案内する。

推奨ルート。

1. `00_作業の流れ.md`
2. `01_最初に投げるprompt.md`
3. `03-1_企画一発生成（project＋image-assets）.md`
4. `08-1_implementation-brief.md`
5. `09_Codex用仕上げレビュー.md`

### 4.5 prompt一覧

利用者が必要な工程のpromptを選べるようにする。

カードに載せる情報。

- ファイル名
- 工程名
- 役割
- 推奨度
- Markdownファイルへのリンク

### 4.6 正本ファイルの考え方

制作時に迷わないよう、正本の役割を説明する。

- `project.md`: 企画、構成、文章、トーンの正本
- `image-assets.md`: 画像、配置、ファイル名、画像内文字の正本
- `implementation-brief.md`: 実装条件、禁止事項、完了条件の短い引き継ぎ
- 実ファイル: `index.html`、CSS、JS、画像

### 4.7 成果物プレビュー

成果物カードは、実際のサイトのファーストビュー画像を使って表示する。

ファーストビュー直下に配置する。

`B:\yuki\repository\slide-archive\archive` 配下の成果物、およびURLに `/slide-archive/archive/` を含む成果物は一覧に反映しない。

例外として、以下の2件だけ掲載する。

- `inage-election`
- `tiktok-cm`

表示ルール。

- 初期表示は3件
- もっと見るで6件ずつ追加
- 追加表示後は閉じるボタンを出す
- 閉じると初期3件表示へ戻す
- 件数が多くなった場合は、ページを伸ばし続けずスクロール枠で対応する

カード項目。

- ファーストビュー画像
- タイトル
- 説明
- 種別ラベル
- 公開URL
- ソース、または関連GitHubリンク

### 4.8 利用開始CTA

最後に、利用開始の導線を置く。

- 全体工程を見る
- prompt一覧へ戻る
- 成果物を見る

## 5. 成果物データ仕様

正本は `data/examples.json`。

```json
{
  "id": "example-id",
  "title": "成果物タイトル",
  "description": "短い説明",
  "label": "HTMLスライド",
  "url": "https://example.com/",
  "sourceUrl": "https://github.com/...",
  "screenshot": "assets/examples/screenshots/example-id.png",
  "featured": true
}
```

## 6. promptデータ仕様

正本は `data/prompts.json`。

```json
{
  "id": "prompt-01",
  "file": "01_最初に投げるprompt.md",
  "title": "初回フィードバック",
  "phase": "企画整理",
  "summary": "最初に課題、確認質問、改善案を出すためのprompt。",
  "href": "prompts/01_最初に投げるprompt.md",
  "starter": true
}
```

## 7. 内部運用: スクリーンショット更新

これは公開LPに載せる内容ではなく、成果物カード用画像を更新するための内部運用である。

成果物画像は手作業で毎回撮らない。PlaywrightでURL一覧を開き、ファーストビューをPNG保存する。

基本フロー。

```text
data/examples.json
↓
tools/capture-first-views.js
↓
assets/examples/screenshots/*.png
↓
git commit / push
↓
GitHub Pagesへ反映
```

GitHub Actionsで週1回の定期実行と手動実行に対応する。

## 8. デザイン方針

現在の青白い資料系から変更してよい。

方向性。

- 閲覧しやすい広告LP寄り
- 見出しは短く強い
- 成果物画像を大きく見せる
- CTAを複数配置する
- 技術説明より「使えそう」と感じる導線を優先する
- 背景は白、淡い青、淡いグレーを中心にする
- 派手な装飾より、カード、ステップ、プレビューで信頼感を出す

## 9. 画像方針

ファーストビューには、最終ヒーロー画像 `assets/images/hero-main.png` を使用する。

今後の画像生成用キャラクター参照として、`assets/images/character-reference.png` を使用する。

方向性。

- 青白ベース
- 派手すぎず、少し遊び心のある広告LP寄り
- ピンク髪の制服キャラクターを右側に配置
- タイトルは画像内に入れる
- 工程説明、矢印、ファイル名、チェックリストは入れない
- キャラクターは基本的にファーストビュー中心で使う

ただし、新LPの初期実装では以下を優先する。

- 成果物ファーストビュー画像は内部運用で自動撮影する
- prompt本文を画像化しない
- URL全文や長文説明を画像内に入れない

## 10. やらないこと

- `slide-archive` と統合しない
- prompt本文をLPに全文掲載しない
- 自動撮影の仕組み自体を公開LPの訴求として見せない
- 表示時にブラウザ内で外部サイトを撮影しない
- 成果物スクリーンショットを手作業前提にしない
- Portfolioページとして設計しない
